Bridge pairing security · 3-layer model

A bridge runs on a managed host alongside an AI agent (Ocean, Merlin, …). Its job is to translate per-turn agent activity into FleetCom dashboard events. The hard question: "how does FleetCom know this 'Ocean' bridge is the real Ocean and not someone with a stolen token claiming to be Ocean?" The answer is layered N-of-3 trust: an attacker must break all three layers below to plant a fake bridge.

Layer 1 · Host bearer token (always on, no toggle)

Every bridge presents the host's bosun bearer token on register. The server hashes it and looks it up.

  • Closes: random attackers on the internet posting fake registrations.
  • Doesn't close: anyone with the token (leaked .env, hijacked process, supply chain) — they can register any bridge.
Layer 2 · Operator approval (auto-approve toggle)

With auto-approve = OFF (the recommended default), every new bridge sits as pending. The header shows PAIR · N. You eyeball the SSH-style fingerprint (e.g. a3:f1:9c:7d:4e:8b:2a:1f) and click APPROVE.

auto-approve Behavior Cost / benefit
OFF (recommended) Each bridge sits pending until you click APPROVE Closes the leaked-token attack: even with the token, attacker can't pair without your eyeballs
ON FleetCom auto-approves once the host token checks out Zero friction; the host token alone is sufficient to register anything
Layer 3 · Gateway co-signature (attest + OOB code)
3a · Cryptographic endorsement (attest): Before registering, the bridge asks its local OpenClaw gateway to sign a statement "yes, this is the bridge claiming agent X with fingerprint Y on host Z." FleetCom verifies the Ed25519 signature against the gateway's pubkey. Status today: FleetCom side fully shipped (FLEET-114). Gateway side pending — until OpenClaw exposes /v1/bridge/sign-registration, this falls through to skipped with an audit row.
3b · Out-of-band confirmation (OOB code): Server mints a 6-digit code on register, salted with the bridge fingerprint. Pushed to the gateway over WebSocket; the gateway is supposed to deliver it through the agent's user channel (Ocean DMs you on Telegram). You read it from the agent and type it into the approval. 5-minute TTL, 5-attempt rate limit, then auto-reject. Status today: FleetCom side fully shipped (FLEET-113). Gateway side pending — until the bridge.confirmation_code RPC handler ships, OOB is dormant and operators must use SKIP OOB on every approval.

Closes: the "host token + gateway both compromised" attack. The attacker still doesn't control the agent's user channel — Ocean's actual conversation with you. The OOB code arrives there or doesn't.

Today's right answer for production

Until the gateway-side OpenClaw RFC ships, this is the maximum useful posture per toggle:

Toggle Value Reason
auto-approve OFF The only working second factor right now.
OOB code OFF Gateway can't deliver codes yet — turning ON forces SKIP OOB on every approval.
attest ON or OFF Equivalent until the global env flag is true. Default ON for forward-compatibility.
+ pubkey leave blank Auto-captured on first connect via TOFU (FLEET-123). Paste only as fallback for rotation/recovery.

The three posture cards above the toggles are the recommended way to set your gateway's stance — one click flips the right combination of flags atomically. Reviewed is the default for production today; Hardened unlocks once the gateway pubkey is captured — FleetCom auto-pins on first connect (TOFU), with manual paste as a fallback. The Advanced toggles disclosure remains for non-canonical combinations.
